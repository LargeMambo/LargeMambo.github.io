为什么需要双仓库？

先搞清楚 Hexo 的工作原理：

默认情况下，hexo deploy 只把 public/ 里的静态文件推到 LargeMambo.github.io。你的文章、配置、主题定制完全没有备份——电脑坏了、误删了、换机器了，全没了。

解决方案： 再建一个私有仓库 Hexo-Blog 专门备份源码，两个仓库各司其职。

两个仓库的分工

仓库 存什么 可见性 怎么写入
LargeMambo/Hexo-Blog 源码：文章、配置、CSS/JS、主题 私有 你手动 git push
LargeMambo/LargeMambo.github.io 静态文件：编译后的 HTML/CSS/JS 公开 hexo deploy 自动推

这两个仓库完全独立，互不影响。hexo deploy 不会动源码仓库，git push 也不会触发页面更新。

第一步：在 GitHub 创建私有源码仓库

  1. 打开 github.com/new
  2. Repository name 填 Hexo-Blog（或任意名字）
  3. 选择 Private（私有，别人看不到你的文章草稿和配置）
  4. 不要勾选 “Add a README file”、”Add .gitignore”、”Choose a license”——保持完全空白
  5. 点击 Create repository

创建完成后，GitHub 会显示一个空仓库页面，复制页面上的仓库地址备用（格式：https://github.com/LargeMambo/Hexo-Blog.git）。

常见问题

Q：主题文件夹是否会被备份？
会，只要主题不是通过 git submodule 安装的（直接复制进 themes/ 目录的主题会被正常备份）。如果主题是 submodule，npm install 后还需要单独 git submodule update --init。

Q：忘记 push 源码，只 deploy 了，会有问题吗？
不会，博客照常运行。只是这次的修改没有备份到私有仓库，下次记得补 push 即可。

Q：能不能用一个仓库的两个分支代替两个仓库？
可以，但不推荐。Pages 仓库是公开的，源码分支会暴露你的草稿、配置和密钥（如果有的话）。双仓库方案更安全、更清晰。
